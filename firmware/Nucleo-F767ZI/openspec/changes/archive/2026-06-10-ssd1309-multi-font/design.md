## Context

`ssd1309-framebuffer` added `ssd1309_draw_char()`/`ssd1309_draw_string()`
backed by a single hard-coded `font5x7[95][5]` table (one byte per glyph
column, bit 0 = top row, fixed 6px advance). There is currently no way to
draw text at a different size; every caller gets the same 5x7 glyphs.

This change generalizes the font representation so the driver can ship
multiple built-in fonts (small/medium/large) and lets callers pick one per
`draw_char`/`draw_string` call.

## Goals / Non-Goals

**Goals:**
- Define a `ssd1309_font_t` descriptor that carries glyph bitmap data plus
  enough metadata (glyph width/height, per-character advance, covered
  character range) for `draw_char`/`draw_string` to render any font.
- Provide three built-in fonts: the existing 5x7 font (unchanged glyph art,
  smallest of the three — fonts below 5px wide are not legibly readable on
  this panel), a new 8x8 "medium" font, and a 10x14 "large" font for
  headings.
- Support glyphs taller than 8px (the large font) without changing the
  framebuffer layout — `draw_char` still goes through `ssd1309_set_pixel()`
  per pixel, so glyph height is not constrained by the GDDRAM page size.
- Keep the change additive at the framebuffer/flush level: no changes to
  `ssd1309_clear`/`fill`/`set_pixel`/`get_pixel`/`flush`/dirty tracking.

**Non-Goals:**
- No proportional/variable-width glyphs — each font remains fixed-width
  (uniform advance per character), just configurable per font.
- No runtime font loading — all fonts are built-in `const` tables in flash.
- No text wrapping, alignment, or multi-line layout helpers.
- No kerning or anti-aliasing/greyscale glyphs (panel is 1bpp).

## Decisions

- **`ssd1309_font_t` descriptor struct**, defined in `ssd1309.h`:
  ```c
  typedef struct {
      const uint8_t *glyphs;   /* flat glyph data, see layout below */
      uint8_t glyph_width;     /* pixel width of one glyph */
      uint8_t glyph_height;    /* pixel height of one glyph */
      uint8_t advance;         /* x distance (px) to next glyph */
      char first_char;         /* first codepoint covered (inclusive) */
      char last_char;          /* last codepoint covered (inclusive) */
  } ssd1309_font_t;
  ```
  Alternatives considered: a `void *` opaque handle with font-specific
  draw callbacks — rejected as over-engineered for three fixed-format
  fonts; a single struct with plain fields is enough and keeps
  `draw_char`/`draw_string` as the only rendering code path.

- **Glyph byte layout (generalizes the existing 5x7 format)**: for a glyph
  of `glyph_width` x `glyph_height`, let
  `rows_per_glyph = (glyph_height + 7) / 8`. Each glyph occupies
  `glyph_width * rows_per_glyph` bytes, ordered column-major: for each
  column `0..glyph_width-1`, `rows_per_glyph` bytes follow, one per
  8-row vertical chunk, bit 0 = top row of that chunk (bits beyond
  `glyph_height` in the last chunk are unused/zero). For `glyph_height <= 8`
  (`rows_per_glyph == 1`) this is byte-for-byte identical to the existing
  `font5x7` layout, so the medium font's glyph data is reused unchanged.
  A glyph for character `c` starts at
  `glyphs[(c - first_char) * glyph_width * rows_per_glyph]`.

- **Three built-in fonts, all covering printable ASCII `0x20`-`0x7E`**:
  - `font5x7` (small): existing 5x7 glyphs, `advance = 6` (unchanged from
    today). Wrapped in a `const ssd1309_font_t font5x7 = { .glyphs = font5x7_glyphs, .glyph_width = 5, .glyph_height = 7, .advance = 6, ... }`;
    the underlying glyph table is renamed (e.g. `font5x7_glyphs`) but its
    bytes are unchanged. This is the smallest built-in font — a dedicated
    sub-5x7 "small" font was tried (3x5) and bench-tested as unreadable, in
    line with general guidance that fonts narrower than ~8px are not legible
    on 128x64 OLED panels, so it was dropped.
  - `font8x8` (medium): 8x8 glyphs (`rows_per_glyph = 1`), `advance = 9`.
  - `font10x14` (large): 10x14 glyphs (`rows_per_glyph = 2`), `advance = 12`.

- **New glyph art generated by scaling the existing 5x7 bitmaps**, not
  hand-drawn from scratch and not copied from a third-party font. A one-off
  Python script renders each 5x7 glyph into a bitmap. The medium font (8x8)
  is produced by corner-aligned nearest-neighbor scaling. The large font
  (10x14) is produced by exact 2x pixel-doubling of the 5x7 bitmap (no
  resampling/distortion) — an earlier nearest-neighbor-scaled 8x13 large
  font was bench-tested and found illegible/ugly, so it was replaced with
  this pixel-doubled 10x14 font. Both are then re-encoded in the
  column-major/bit-chunk layout above. This keeps all glyph art self-derived
  (avoiding the licensing concern noted in the framebuffer change) while
  avoiding ~190 glyphs of manual bitmap authoring. The scripts and their
  intermediate output are scratch artifacts (not committed); only the
  resulting `static const uint8_t` tables go into `ssd1309.c`.

- **API change**: `ssd1309_draw_char()` and `ssd1309_draw_string()` gain a
  `const ssd1309_font_t *font` parameter, inserted right after `p`:
  ```c
  void ssd1309_draw_char(ssd1309_t *p, const ssd1309_font_t *font, int16_t x, int16_t y, char c, uint8_t color);
  void ssd1309_draw_string(ssd1309_t *p, const ssd1309_font_t *font, int16_t x, int16_t y, const char *str, uint8_t color);
  ```
  This is a **BREAKING** change to both functions; the only existing call
  site (`app_init()` in `app.c`) is updated to pass `&font5x7`.

- **`draw_char` rendering loop**: for `c` outside
  `[font->first_char, font->last_char]`, render blank (no pixels touched),
  same as today. Otherwise, for each column `0..glyph_width-1` and each
  chunk `0..rows_per_glyph-1`, read the glyph byte and, for each set bit
  `b` where `chunk*8+b < glyph_height`, call
  `ssd1309_set_pixel(p, x+col, y+chunk*8+b, color)`. This is a direct
  generalization of the current loop (which is the `rows_per_glyph == 1`
  case).

- **`draw_string`** advances `x` by `font->advance` per character, calling
  `ssd1309_draw_char()` with the same `font`. No wrapping, matching the
  current behavior.

## Risks / Trade-offs

- **Flash usage grows ~3x for font data** (three tables instead of one,
  plus the larger 10x14 table). → Mitigation: all tables are `const` (flash,
  not RAM); STM32F767ZI has 2MB flash, this is negligible.
- **Algorithmically scaled glyphs may look cruder than hand-tuned bitmaps**,
  particularly the nearest-neighbor-scaled 8x8 medium font where some
  characters may look blocky/uneven. → Mitigation: bench-check legibility
  during verification; if specific glyphs are unusable, hand-patch just
  those entries in the generated table (acceptable one-off touch-up, not a
  redesign). The large font avoids this risk entirely by using exact 2x
  pixel-doubling instead of resampling.
- **Breaking API change** to `draw_char`/`draw_string` requires updating the
  one existing call site. → Mitigation: single call site in `app.c`,
  updated as part of this change (task 5).
